Perguntas e Respostas Git

  1. O que são operações de “Commit”, “Push” e “Pop” efetuadas num repositório?

    R: São operações que permitem a alteração de um repositório em termo de ficheiros existentes.

    • Commit: Este comando permite que alterações locais do repositório sejam registadas. É boa prática adicionar uma mensagem através do comando -m. Esta mensagem deve ser breve e descrever as alterações realizadas localmente, o que é útil no controle de versões, pois permite uma rápida identificação das alterações através do histórico de commits.
    • Push: Este comando permite a efetivação das alterações locais registadas pelo commit. Ou seja, ficheiros criados, modificados ou deletados são replicados no repositório remoto. É possível especificar qual branch será o alvo da sincronização. Essas alterações são visíveis apenas para os utilizadores que possuem a versão atualizada do repositório, ou seja, clonando-o ou executando o comando pull.
    • Pop: Este comando permite a restauração de alterações armazenadas no stash. O stash é utilizado para guardar temporariamente alterações realizadas no repositório, mas que não devem ser sincronizadas de imediato ao fazer um commit. Enquanto o stash desfaz temporariamente as alterações no repositório local, o pop permite que estas voltem a ter efeito localmente.

  2. Quais são as vantagens de usar Branches num repositório git?

    R: O uso de branches permite o desenvolvimento paralelo sem riscos de sobreposição de alterações. Trata-se de "ramificações" do repositório remoto. Por defeito, existe sempre um branch principal, chamado master ou main. Outras ramificações permitem o desenvolvimento e gestão de ficheiros de forma isolada. Isto é possível porque cada branch possui o seu próprio código, e sincronizações através de push afetam especificamente o branch desejado, sem alterar o main. Além disso, cada branch tem seu próprio controle de versões e histórico. Isso permite testes, identificação de bugs, e desenvolvimento de novas funcionalidades de forma paralela, sem afetar o código principal.

  3. Nos Merges de conteúdos resultantes de um “Pop” podem resultar em conflitos. Em que consistem e como podem ser resolvidos esses conflitos?

    R: Conflitos em um merge ocorrem quando duas ou mais alterações são feitas no mesmo ficheiro, mais especificamente na mesma linha ou bloco de código, e o Git é incapaz de identificar qual versão deve prevalecer. Um conflito comum acontece quando um utilizador faz alterações locais, e outro utilizador altera o mesmo trecho de código no repositório remoto. Isso pode causar um conflito no momento de integrar as mudanças (merge), porque o Git não consegue decidir qual versão é a correta.

    No caso do pop, ao restaurar alterações guardadas no stash, pode haver um conflito se o código restaurado entrar em conflito com modificações feitas por outro utilizador ou se alterações locais tiverem sido feitas no mesmo trecho.

    Quando ocorre um conflito, o Git sinaliza ficheiros afetados. É demarcado no código da seguinte forma:

    <<<<<<< HEAD
(versão local)
=======
(versão nova)
>>>>>>>

    Uma forma de resolução é abrir o ficheiro sinalizado e decidir qual o código correto a considerar, tal como remover as marcações. Uma vez que o ficheiro esteja corrigido, deve ser sincronizado com o repositório remoto.

  4. "Clone" e "Fork" são formas diferentes de replicar um repositório Git. Quais as diferenças entre eles?

    R: Ambos os comandos criam uma cópia local do repositório, a diferença está no propósito desta cópia.

    • Clone: O comando git clone é utilizado quando o objetivo é contribuir diretamente com o repositório clonado. Ele cria uma cópia exata do repositório no ambiente local do utilizador, mantendo uma conexão direta com o repositório original. Isso significa que operações como pull podem trazer mudanças do repositório remoto para o local, e operações como push permitem enviar mudanças de volta ao repositório original, alterando-o diretamente (desde que o utilizador tenha permissão). Essa abordagem é ideal para colaboradores que têm acesso para fazer mudanças diretas no projeto.
    • Fork: Utilizado com o intuito de criar uma cópia independente do repositório original. A cópia local fica isolada do original, permitindo que o utilizador faça modificações sem afetar o projeto principal. Se o utilizador quiser que suas alterações sejam aplicadas ao repositório original, ele pode enviar um pull request. Esta abordagem é comum em projetos "open source".
